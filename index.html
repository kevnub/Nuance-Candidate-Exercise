Xfinity Digital Assistant - Payment Process

By: Kevin Tran

UX case study to design a user flow and interface in which users utilize a virtual assistant in a chat platform to pay a bill.

View the demo on Invision

The Problem

55% of Xfinity’s chat requests are due to customers trying to view/pay their bill or schedule a technician to set up internet/cable.

The Solution

Xfinity will make use of virtual assistants to handle bill payments and technician scheduling issues, with all other requests being routed to a live agent.

The Objective

Facilitate the bill payment process for the user within the confines of Xfinity’s chat platform.

Visions

1. How might we utilize different forms of input besides dialog to efficiently guide the user to their goal?

2. How might we decrease the amount of actions that the user will have to perform?

Competitive Research

Looking at competitors such as Spectrum, AT&T, and Cox, it is clear that there is a standard in virtual agent/assistant dialog flows with clear call to actions and similar user interfaces.

Dialog Flow

To design the dialog flow for a payment process, we need to gather the information needed from the user to complete a payment.

Information that we'll need from the user:

- Credit/debit card number
        - Name on card
        - Expiration Date
        - Security Code
- Billing Address
        - Name
        - Address
        - Phone Number

Pure Dialog vs Form Input

A challenge that arose was deciding whether to implement a payment process that relied purely on user and chat dialog boxes or guiding the user to a payment form that they could fill out.

Pros of pure dialog

  • User won't be taken to another screen

  • Quick payment process if all payment information is already on file

Cons of pure dialog

  • Information can get lost in chat bubbles

  • Difficult to make multiple edits to user information

  • User won't know how many stages there are to complete a payment

Pros of using forms

  • All information in one place

  • Easily editable information

  • Simple to keep track of stages in the payment process

Cons of using forms

  • Requires a different screen or window to appear

  • Defeats the purpose of the ease of using a chat messenger

A combination of dialog and forms

That solution I'm presenting is a hybrid between a user filling out forms and confirming existing payment information. Forms will be presented to the user only when necessary. I.e., if a user needs to log in or edit their payment/billing information. Otherwise, the virtual agent will only need a simple confirmation of user information that should already be on file.

If user information is already on file, it will be presented to the user and simply ask for a confirmation that it is correct. In this case, the user will only have to input the security code of their card; single button presses will be sufficient to confirm the rest of the information.

In the event that the user will need to input or update their payment and billing information, a form will slide up within the chat window where the user can easily complete the payment process.
